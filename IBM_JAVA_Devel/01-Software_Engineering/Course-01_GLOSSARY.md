# Software Engineering Glossary

This alphabetised glossary contains terms from this comprehensive software engineering course, including industry-recognised terminology essential for professional practice, participation in user groups, and advanced certificate programmes.

---

| Term | Definition | Guide |
|------|-----------|-------|
| **2-tier** | A computing model in which the server hosts, delivers, and manages most resources and services delivered to the client. | 04-007 |
| **3-tier** | Architecture organising applications into three logical and physical computing tiers: presentation tier (user interface), application tier (business logic processing), and data tier (data storage and management). | 04-007 |
| **Agile** | An iterative software development method where teams work in cycles (sprints). Unit testing occurs in each sprint, and working code is released at sprint meetings for stakeholder feedback. | 01-005 |
| **Alpha release** | The first functioning version of a system released to a select group of stakeholders. | 01-006 |
| **API gateway** | Component routing API requests from clients to services; orchestration handles inter-service communication. | 04-007 |
| **Application deployment environment** | The combination of hardware and software resources required to run an application. | 04-008 |
| **Application Programming Interface (API)** | Code enabling two programmes to communicate, defining how applications share and modify each other's data. | 02-003 |
| **Array** | Fixed-size collection of same-type elements stored sequentially, indexed from zero. | 03-001 |
| **Assembly language** | Low-level programming language using symbols representing machine code. | 03-006 |
| **Attributes** | An object's properties and data. | 04-003 |
| **Backend developers** | Engineers handling server-side functionality before code and data transmission to clients. | 02-001 |
| **Behavioral models** | Diagrams and representations describing system behaviour without explaining implementation details. | 04-002 |
| **Beta release** | Limited-release version distributed to stakeholders outside the developing organisation. | 01-006 |
| **Black-box testing** | Testing method examining inputs and outputs without reviewing source code or internal structure. | 01-008 |
| **Blackboard** | Architectural pattern with shared knowledge repository, independent knowledge sources, and control component for iterative problem-solving. | 04-005 |
| **Boolean expression** | Programming statement with two values: "true" or "false." | 03-003 |
| **Branching** | Logic enabling programmes to execute different instruction sets based on condition evaluation during execution. | 03-003 |
| **Build automation servers** | Execute build automation utilities on scheduled or triggered basis. | 02-004 |
| **Build automation utilities** | Generate build artefacts (executables) by compiling and linking source code. | 02-004 |
| **Business logic** | Application rules dictating transaction results and database data operations (read, write, retrieval). | 04-009 |
| **Canarying** | Deployment process rolling out new code to test functionality before full release, with rollback capability. | 04-010 |
| **Cascading Stylesheets (CSS)** | Standard method for front-end developers defining, applying, and managing website style characteristics (colours, fonts, layouts, designs). | 02-002 |
| **Chief Technology Officer (CTO)** | Organisational technology arm leader overseeing research and development, systems, infrastructure, and technology evaluation for competitive advantage. | 05-007 |
| **Class** | Generic blueprint for objects containing generic attributes (properties and methods). | 04-003 |
| **Class diagram** | Structural UML diagram showing class relationships, attributes, and interactions in object-oriented design. | 04-003 |
| **Client-Server** | Alternative term for 2-tier architecture. | 04-007 |
| **Code of Ethics** | Professional standards framework (eight principles) developed by IEEE-CS and ACM for software engineer accountability. | 05-010 |
| **Cohesion** | Degree to which functionally related elements are grouped together within modules. | 04-002 |
| **Compiled language** | Programming language using compiler to convert source code to machine code, then executable programme file. | 03-005 |
| **Component** | Individual unit of encapsulated functionality serving as application part alongside other components. | 04-006 |
| **Component-based architecture** | Design approach decomposing systems into logical, loosely coupled independent components providing higher abstraction than OOP alone. | 04-006 |
| **Conditional statement** | "If-then" statement enabling branching logic. | 03-003 |
| **Constant** | Data item with unchanging value within a programme. | 03-001 |
| **Controller** | Component type determining which other components to invoke for particular events, controlling data flow between components. | 04-006 |
| **Controller-Responder** | Architectural pattern separating request handling (controller) from action execution (responder). | 04-005 |
| **Continuous Integration / Continuous Delivery (CI/CD)** | DevOps practices enabling reliable frequent code changes. CI ensures component compatibility; CD deploys tested code to staging/testing environments. | 02-004 |

| **Create, Read, Update, Delete (CRUD)** | Shorthand for common database query types. | 03-006 |
| **Data access object (DAO)** | Component providing database interface, enabling application database switching without code modification. | 04-006 |
| **Database** | Collection of related data stored on computer, accessible through various methods and controlled by DBMS. | 04-009 |
| **Database Management System (DBMS)** | Software controlling database access by connecting databases to users and programmes. | 04-009 |
| **Debugging** | Process identifying and resolving code defects; equally important as writing code. | 04-010 |
| **Deployment** | Distribution of completed software solutions to users via shell scripting, containers, CI/CD tools. | 04-008 |

| **Dynamic content** | Content generated each time requested by client. | 02-001 |
| **Encapsulation** | Bundling component data and methods, hiding internal state and specific implementation. | 04-006 |
| **Engineering Manager** | Role ensuring team support and encouragement, facilitating career progression. | 05-007 |
| **Event** | Anything resulting in state change; action triggered by end-user or programme component. | 04-007 |
| **Event-driven** | Architectural pattern where producers publish events and consumers subscribe; decouples producers from consumers. | 04-007 |
| **Event notification** | Message generated by triggering event, sent to listening consumer. | 04-007 |
| **Extensibility** | Ability to add behaviour to components without modifying other components. | 04-006 |
| **Firewall** | Security device monitoring network traffic, permitting/blocking data based on security rules; barrier against viruses, malware, hackers. | 04-008 |

| **Frontend developers** | Engineers handling client-side functionality; everything users see and interact with. | 02-001 |
| **Full-stack developers** | Engineers with skills, knowledge, and experience in both frontend and backend environments. | 02-001 |
| **Full-stack engineers** | Create entire application solutions (UI and back-end functionality) combining front-end and back-end skills. | 05-008 |
| **Function** | Structured, standalone, reusable code performing single specific action; accepts input, processes data, returns output. | 03-002 |
| **Functional testing** | Black box testing verifying inputs and outputs against specifications. | 01-008 |
| **Games developer/engineer** | Write gaming software for PCs, smartphones, browsers, games consoles; collaborate with graphic artists and designers. | 05-008 |
| **General availability** | Stable software version intended for all users. | 01-006 |

| **Hard skills** | Practical, technical, quantifiable skills needed for specific roles; learned through education or experience. | 05-003 |
| **Hybrid Cloud** | Mix of public and private clouds working seamlessly together, optimising cost, security, scalability, and flexibility. | 04-008 |

| **Information developer** | Alternative term for technical writer. | 01-002 |
